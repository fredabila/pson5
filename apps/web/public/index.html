01 · The Standard

A file format that keeps fact, inference, and prediction separate.

Most agent memory collapses three very different claims into one blob of text. .pson doesn't. Direct user facts, model-derived traits, and scenario-specific predictions live in distinct layers — each with confidence, evidence, and decay policy attached.

Observed Direct user statements and normalized answers.
confidence 1.0
Inferred Traits, heuristics, and states derived from evidence.
half-life 14d
Simulated Scenario predictions, not durable facts.
reasoning trace
profile.pson 5.0 
{
  "pson_version": "5.0",
  "profile_id":  "pson_1776913414584",
  "consent": { "granted": true, "scopes": ["ai:use"] },

  // what the user said
  "layers.observed": {
    "core.facts.deadline_effect": "mixed",
    "core.facts.planning_style":  "structured"
  },

  // what the model inferred
  "layers.inferred": {
    "heuristic": "deadline_driven_activation",
    "confidence": 0.78,
    "decay_policy": { "half_life_days": 14 }
  },

  // what the simulator predicts
  "layers.simulated": {
    "prediction": "delayed_start",
    "reasoning": ["path via graph"],
    "caveats": ["sparse scenario data"]
  }
}
02 · The Pipeline

From raw signal to agent-safe context.

PSON5 is not a single box. It is a pipeline: six specialized engines that each own one concern, communicate through the .pson contract, and compose cleanly behind a unified SDK, API, CLI, and MCP server.

01

Acquisition

Adaptive question selection with confidence-gap scoring, contradiction flags, and fatigue detection.

@pson5/acquisition
02

Modeling

Rule-based and provider-backed derivation of traits and heuristics with evidence and decay.

@pson5/modeling
03

State

Transient conditions with time-decay and runtime trigger evaluation against observed facts.

@pson5/state
04

Graph

Knowledge graph with k-hop traversal and path-based prediction explanations.

@pson5/graph
05

Simulation

Scenario-specific predictions with reasoning, caveats, and alternatives — never promoted to fact.

@pson5/simulation
06

Projection

Agent-safe context with explicit redaction notes, consent gates, and relevance ranking.

@pson5/agent-context
03 · The Three Layers

One model. Three epistemic states.

Collapsing fact, inference, and prediction into one blob is how agent memory goes wrong. PSON5 keeps them visibly separate, so an LLM can treat each with the weight it deserves.

01 · Observed

What the user said.

Direct statements and normalized answers. High certainty. The anchor the rest of the system is allowed to reason from.

layers.observed.core.facts
deadline_effect = "mixed"
planning_style = "structured"
learning_mode = "doing"
02 · Inferred

What the model thinks.

Traits, heuristics, and states derived from evidence. Always carries confidence, evidence references, and a decay policy that ages it out over time.

layers.inferred.heuristics
deadline_driven_activation
  confidence: 0.78
  half_life: 14d
03 · Simulated

What the engine predicts.

Scenario-specific estimates with reasoning trace and caveats. Never written back as observed fact. A hypothesis, not a record.

04 · Surfaces

Drop PSON5 into any runtime your agent already lives in.

Same engine, same .pson contract, four transports. Use the SDK from your backend, the HTTP API from anywhere, the CLI for local workflows, or MCP when a framework expects it.

  • @pson5/sdkTypeScript, server-firstdirect
  • /v1/pson/*HTTP, auth-gated, auditedremote
  • psonCLI with --json for scriptinglocal
  • /v1/mcp · mcp-stdioJSON-RPC for agent frameworksmcp
// Same engine, direct call. One context, already redacted.
import { PsonClient } from "@pson5/sdk";

const pson = new PsonClient();

const context = await pson.getAgentContext("pson_123", {
  intent:              "help the user plan a deadline-sensitive task",
  include_predictions: true,
  max_items:           6,
  min_confidence:      0.6
});

// context.personal_data — relevance-ranked
// context.redaction_notes — why anything was filtered
// context.constraints.restricted_fields — still honoured
# HTTP. Auth-gated. Per-request id. Redacted for non-admin.
POST /v1/pson/agent-context HTTP/1.1
content-type: application/json
authorization: Bearer <jwt>
x-pson-tenant-id: tenant_acme

{
  "profile_id": "pson_123",
  "intent":     "help the user plan",
  "max_items":  6
}

# response includes redaction_notes + constraints
# header: x-pson-request-id: req_<uuid>
# Local profile.  --json makes it scriptable.
$ pson init study_user --store .pson5-store

05 · For Agents

Built for agents, not against them.

Agents get a standardized tool contract, a ready-to-load Claude skill, and MCP transport. They never need to read the raw profile — the projection already enforces consent, confidence, and restricted fields.

Agents building PSON5 must preserve this distinction throughout the codebase: what the user said, what the system inferred, what the simulator predicts.
  1. 01
    Probabilistic over absolute Every derived claim carries confidence. Nothing is inferred as fact.
  2. 02
    Consent-first data collection Scopes gate acquisition. Restricted fields follow the profile out the door.
  3. 03
    Explainability as a deliverable Predictions carry reasoning traces. Redactions carry reason codes.
  4. 04
    Portability of the profile .pson is exportable, validatable, and agnostic to your runtime.
  5. 05
    Confidence that decays Yesterday's inference is weaker than today's. Evidence ages.
