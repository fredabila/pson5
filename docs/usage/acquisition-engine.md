# @pson5/acquisition-engine

> Adaptive question flow, session persistence, and answer writeback. The entry point for adding evidence to a profile.

## Install

```bash
npm install @pson5/acquisition-engine
```

## What this package does

The acquisition engine is the "learning" surface of PSON5. It owns:

- **Question registry** — the built-in core / education / productivity questions plus any domain modules registered via `saveDomainModules(...)`.
- **Learning sessions** — a `LearningSessionState` persisted at `<store>/sessions/<session_id>.json` that tracks which questions have been asked, answered, generated by the provider, and which confidence gaps remain.
- **Adaptive selection** — when a provider is configured, `deriveAdaptiveQuestion` routes the next question via the provider (with rewrites and a natural answer-style hint). With no provider, deterministic depth filtering plus unseen-question selection.
- **Answer submission and pipeline trigger** — on `submitLearningAnswers(...)` the engine writes the normalized observed answer, then triggers the modeling → state → graph pipeline and bumps the profile revision.

## Exports

```ts
import {
  getBuiltInQuestionRegistry,
  getQuestionRegistry,
  listDomainModules,
  saveDomainModules,
  getQuestionsForDomains,
  getNextQuestions,
  submitLearningAnswers,
  acquisitionEngineStatus
} from "@pson5/acquisition-engine";
```

| Export | Returns | Purpose |
| --- | --- | --- |
| `getBuiltInQuestionRegistry()` | `QuestionDefinition[]` | Synchronous snapshot of the built-in core / productivity / education questions. |
| `getQuestionRegistry(options?)` | `Promise<QuestionDefinition[]>` | Built-in registry merged with any domain modules saved under the store. |
| `listDomainModules(options?)` | `Promise<DomainModuleDefinition[]>` | Read domain modules from `<store>/config/domains.json`. |
| `saveDomainModules(modules, options?)` | `Promise<{ path, count }>` | Persist domain modules; used by admin tooling. |
| `getQuestionsForDomains(registry, domains, depth)` | `QuestionDefinition[]` | Depth-filtered candidate set from a registry. |
| `getNextQuestions(profileId, input, options?)` | `Promise<LearningSessionResult>` | Open or resume a session, pick the next question(s), persist the session, return `{ session, questions }`. |
| `submitLearningAnswers(input, options?)` | `Promise<LearnResult>` | Persist answers, run modeling / state / graph, save the profile, update the session. |

## Usage

### One-question-at-a-time flow

```ts
import { getNextQuestions, submitLearningAnswers } from "@pson5/acquisition-engine";

const store = { rootDir: ".pson5-store" };

const first = await getNextQuestions("pson_123", { limit: 1 }, store);
const q = first.questions[0];
// show q.prompt to the user (q.answer_style_hint describes how to answer)

const result = await submitLearningAnswers(
  {
    profile_id: "pson_123",
    session_id: first.session.session_id,
    answers: [{ question_id: q.id, value: "delay_start" }],
    options: { return_next_questions: true, next_question_limit: 1 }
  },
  store
);

// result.profile is the updated PsonProfile (revision bumped)
// result.session carries contradiction_flags, fatigue_score, confidence_gaps,
// and stop_reason if the engine decided to pause
// result.next_questions is the follow-up, or [] if the session should stop
```

### Wiring a custom domain

Domain modules are saved under `<store>/config/domains.json` and picked up automatically by `getQuestionRegistry`. Each module declares its own questions:

```ts
import { saveDomainModules } from "@pson5/acquisition-engine";

await saveDomainModules(
  [
    {
      id: "finance",
      version: "1.0.0",
      description: "Short finance-risk module",
      questions: [
        {
          id: "fin_risk_tolerance",
          domain: "finance",
          prompt: "How do you feel about short-term volatility?",
          type: "single_choice",
          depth: "standard",
          sensitivity: "standard",
          information_targets: ["risk_tolerance"],
          follow_up_rules: [],
          choices: [
            { value: "low", label: "I avoid it." },
            { value: "medium", label: "I tolerate it." },
            { value: "high", label: "I seek it." }
          ]
        }
      ]
    }
  ],
  { rootDir: ".pson5-store" }
);
```

## Session intelligence

Every session surfaces four signals that PSON5 uses internally to decide when to stop asking:

- `fatigue_score` — 0 → 1, combines total asked count and unanswered share. The engine pauses early when fatigue ≥ 0.78 and remaining uncertainty is low.
- `confidence_gaps` — information targets from active domains with no observed / inferred signal yet.
- `contradiction_flags` — diffs detected when a new answer overwrites an existing observed fact for the same target.
- `stop_reason` — populated when the engine (or provider) decides to pause. `null` when the session is still active.

Agents should respect all four. Keep asking until `stop_reason` is set, or until `confidence_gaps` collapses for the active domains.

## Key concepts

- **Answers never merge silently.** If a new answer contradicts an existing observed fact, the change is recorded and tagged in `session.contradiction_flags`. The profile still takes the new value, but the flag remains so the caller can surface the conflict.
- **Provider-generated questions are tagged.** When the provider rewrites a prompt or generates a follow-up, the resulting `QuestionDefinition` has `generated_by: "provider"`, a `source_question_id` pointing back to the approved target, and a `generation_rationale`.
- **The pipeline runs on every submit.** Observed answer → modeling-engine → state-engine → graph-engine → serialization-engine save. If any engine throws, the answer is not persisted.
- **Sessions are portable.** Move the `<store>/sessions/` directory and the session resumes on any PSON5 deployment pointing at the same store.
